API Testing Basics

Introduction to API

API stands for Application Programming Interface. In simple terms, an API acts as an interface or communication protocol between the client (frontend) and the server (backend) in a software system.

The role of an API is to simplify the interaction between different software components, often written in different programming languages or frameworks, by providing a standard way to exchange data.

Why Do We Need APIs?

  • APIs solve the problem of different technologies on frontend and backend needing to communicate. For example:
    • Frontend might be developed using Angular (TypeScript).
    • Backend might be developed in Java, with a database storing data.
  • APIs provide a language-agnostic, standard way (usually over HTTP) for clients and servers to talk.
  • APIs enable third-party integrations, such as letting hotels.com display availability from Marriott Hotels without sharing backend code.

How APIs Work: Basic Flow

Consider a hotel booking system:

  • The frontend collects booking details (e.g., check-in, check-out dates).
  • These details are sent to an API on the backend server, which processes the request.
  • The backend creates or fetches the reservation in the database.
  • The API responds to the frontend with confirmation details (e.g., reservation ID).

Common API Design Constraints

  • APIs are exposed as endpoints (URLs) on servers.
  • Communication protocol used is generally HTTP.
  • Data format for request and response is commonly JSON or XML.
  • APIs abstract backend methods to be language-neutral and secure.

Common HTTP Methods in REST APIs

Method Purpose Example Usage
GET Retrieve data from server Get details of an existing reservation by ID
POST Create a new resource on server Submit booking details to create a reservation
PUT Update a resource completely or partially Update the address of a booking/reservation
DELETE Remove a resource Delete a booking or place

Resource & Parameter Concepts

  • Resource: Represents a specific API functionality or data (e.g., "maps" in Google Maps API).
  • Path Parameters: Used to specify or drill down to a specific resource item (e.g., /orders/{orderId}).
  • Query Parameters: Used to filter or sort results, added after '?' in URL (e.g., ?key=value).

Headers and Cookies

Headers and cookies carry metadata or additional information required by the API, such as:

  • Authentication or authorization details (tokens, API keys).
  • Content-Type indicating data format being sent (e.g., application/json).
  • Session cookies for maintaining user state.

Manual API Testing with Postman: Example

Consider a Google Maps-like API with multiple endpoints for place management:

  • Add Place (POST): Submits data about a new place (name, location, phone, etc.) in JSON format.
  • Get Place (GET): Retrieves details of a place by its unique ID.
  • Update Place (PUT): Updates existing place details like address.
  • Delete Place (DELETE): Removes a place from the records.

Each API requires a contract that defines:

  • Base URL
  • Resource paths
  • HTTP methods
  • Request body format and parameters
  • Expected responses

Steps to Test in Postman

  1. Set the full endpoint URL (base URL + resource path).
  2. Select HTTP method (e.g., POST for adding place).
  3. Add authentication keys or query parameters if required.
  4. Set request body (usually JSON) in "Raw" tab with application/json header.
  5. Click “Send” and verify the response status code (200 OK indicates success).
  6. Use the response data (e.g., place ID) for subsequent API calls like GET or DELETE.

This method allows thorough manual testing to verify the API behaves as expected before automating tests.

These core API concepts and testing practices lay the foundation for advanced API automation using tools like Postman or Rest Assured, covered in subsequent lectures.
