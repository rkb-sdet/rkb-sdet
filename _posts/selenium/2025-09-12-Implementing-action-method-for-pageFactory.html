Writing Action Methods in Page Objects (Selenium POM)

Instead of performing UI actions (like login) step-by-step in your tests, you can combine multiple steps into a single action method in your Page Object class. This increases readability and keeps all element logic and interaction within the related page file.

Encapsulate Actions as Methods

For example, login involves entering the email, password, and clicking the login button. All three lines can go into one reusable method like loginApplication(email, password) in your LandingPage class.

Page Object Example with Action Methods

package rahulshettyacademy.pageobjects;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.FindBy;
import org.openqa.selenium.support.PageFactory;

public class LandingPage {
  WebDriver driver;

  // Page Elements
  @FindBy(id = "userEmail")
  WebElement userEmail;

  @FindBy(id = "userPassword")
  WebElement password;

  @FindBy(id = "login")
  WebElement submit;

  // Constructor initializes elements
  public LandingPage(WebDriver driver) {
    this.driver = driver;
    PageFactory.initElements(driver, this);
  }

  // Action method for login scenario
  public void loginApplication(String email, String pwd) {
    userEmail.sendKeys(email);
    password.sendKeys(pwd);
    submit.click();
  }

  // Method for navigating to the landing page
  public void goTo() {
    driver.get("https://rahulshettyacademy.com/client");
  }
}

Cleaner Usage in Tests

WebDriver driver = new ChromeDriver();
LandingPage landingPage = new LandingPage(driver);

// Navigate, then login with one method call
landingPage.goTo();
landingPage.loginApplication("your_email@example.com", "your_password");

Your test code becomes more readable and maintainable. Anyone reading your test instantly knows the sequence of UI actions being performed.

Page Object Example with List Elements

For lists of elements (like products), declare them as List<WebElement> using @FindBy annotation, making it easy to loop or filter in the action methods. 

package rahulshettyacademy.pageobjects;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.FindBy;
import org.openqa.selenium.support.PageFactory;
import java.util.List;

public class ProductCatalogue {
  WebDriver driver;

  // List for all products on the page
  @FindBy(css = ".mb-3")
  List<WebElement> products;

  public ProductCatalogue(WebDriver driver) {
    this.driver = driver;
    PageFactory.initElements(driver, this);
  }

  // Example action method: get product list
  public List<WebElement> getProductList() {
    return products;
  }
}
  • Action methods group several steps for single scenarios (login, search, checkout).
  • Test scripts only call these methods and supply relevant data (no hardcoding in page objects).
  • List element annotation enables easy collection handling (loop, filter, click).
  • Common utilities like explicit waits should be centralized in base classes for reusability.
Tip: Page object classes should only hold elements and actions. Pass data from your tests; never hardcode data in page objects.

Summary

Page Object Model with action methods enables scalable, readable automation design. Encapsulate logic per page, pass data from tests, and use reusable utility methods in a shared base class for efficiency.
